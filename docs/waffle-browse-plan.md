# Waffle Browse 개발 계획 분석

## 1. 핵심 방향

Waffle Browse는 파일 탐색기를 처음부터 다시 구현하는 앱이 아니다. 파일과 폴더를 표시하는 핵심 영역은 Windows Shell/Explorer View를 그대로 사용하고, 그 주변에 멀티 패널, 탭, 검색, 즐겨찾기, 작업공간 기능을 붙이는 Windows 전용 래퍼 앱으로 설계한다.

이 방향의 핵심 장점은 다음과 같다.

- Windows 파일 탐색기와 동일한 파일 표시 경험을 유지할 수 있다.
- 아이콘, 썸네일, 정렬, 컬럼, 보기 방식, 드래그 앤 드롭, 우클릭 메뉴 같은 Shell 기능을 직접 재구현하지 않아도 된다.
- Waffle Browse는 파일 표시 자체보다 다중 패널 워크플로, 빠른 검색, 작업공간 저장/복원에 집중할 수 있다.

반대로 이 설계는 Windows Shell COM 연동에 강하게 의존한다. 따라서 초기 기술 검증 단계에서 `IExplorerBrowser` 호스팅이 안정적으로 가능한지 확인하는 것이 프로젝트의 성패를 가르는 핵심이다.

## 2. 목표 구조

```text
Waffle Browse App
├─ 자체 UI
│  ├─ 상단 툴바
│  ├─ 탭
│  ├─ 1~4패널 레이아웃
│  ├─ 탭 드래그 기반 패널 도킹
│  ├─ 빠른 검색
│  ├─ 즐겨찾기
│  └─ 작업공간 저장/불러오기
│
└─ Explorer Shell Panel
   ├─ 파일/폴더 표시
   ├─ Windows 아이콘/썸네일
   ├─ 정렬/컬럼/보기 방식
   ├─ 드래그 앤 드롭
   └─ Windows 우클릭 메뉴
```

Waffle Browse의 자체 UI는 사용자가 여러 위치를 빠르게 오가고 작업 단위를 저장할 수 있게 하는 영역이다. Explorer Shell Panel은 실제 파일 목록을 표시하고 Windows Shell 기능을 제공하는 영역이다.

## 3. 기술 선택

### 3.1 추천 스택

1. C# WPF + Win32/Shell COM interop
2. WinUI 3
3. Avalonia

초기 구현은 C# WPF가 가장 현실적이다. `IExplorerBrowser` 같은 네이티브 HWND/COM 컴포넌트를 앱 내부 패널에 호스팅해야 하므로, Windows 데스크톱 앱 모델과의 호환성이 중요하다. WPF는 Windows 전용 앱이라는 목표와도 잘 맞고, Win32 interop 사례와 안정성이 상대적으로 풍부하다.

WinUI 3는 Windows 11 스타일과 최신 UI 구성에는 유리하지만, Shell COM 호스팅과 패키징에서 불확실성이 커질 수 있다. Avalonia는 크로스플랫폼 장점이 있지만, Waffle Browse는 Windows Shell 기능에 강하게 묶여 있으므로 첫 선택으로는 비효율적이다.

### 3.2 핵심 기술 리스크

- `IExplorerBrowser`를 WPF 패널 안에 안정적으로 호스팅할 수 있는지
- 폴더 경로 이동과 탐색 히스토리를 앱 상태와 동기화할 수 있는지
- Windows 우클릭 메뉴가 기존 탐색기와 동일한 Shell 메뉴로 표시되는지
- 1~4개의 Shell 패널을 동시에 띄웠을 때 포커스, 리사이즈, DPI, 성능 문제가 없는지
- 탭/패널 전환 시 Shell View의 수명 주기와 리소스 해제가 안정적인지
- 탭 드래그로 패널 구조를 바꿀 때 Shell View의 포커스와 리사이즈가 안정적인지

## 4. 단계별 개발 계획

### 4.1 0단계: 프로젝트 생성

목표는 개발 가능한 기본 골격을 만드는 것이다.

- 저장소명: `waffle-browse`
- 앱 이름: `Waffle Browse`
- 기본 WPF 프로젝트 생성
- Shell interop 실험용 프로젝트 구조 준비
- 탭, 패널, 레이아웃 상태를 분리한 기본 모델 설계
- 향후 UI, Shell Panel, Workspace, Search 기능을 분리할 수 있는 기본 폴더 구조 정의

이 단계에서는 기능 완성보다 실험과 확장에 적합한 구조를 만드는 것이 중요하다.

### 4.2 1단계: Shell Panel PoC

가장 먼저 검증해야 하는 단계다. 이 단계가 실패하면 Waffle Browse의 전체 방향을 재조정해야 한다.

검증 항목:

- 앱 패널 안에 `IExplorerBrowser` 표시
- 지정한 폴더 경로로 이동
- 기본 파일/폴더 목록 표시
- Windows 우클릭 메뉴 표시
- Windows의 추가 옵션 표시와 유사한 Shell 메뉴 동작 확인
- 창 크기 변경 시 Shell View 리사이즈 확인
- 단일 패널에서 포커스 이동과 키보드 입력 확인

완료 기준:

- 한 개의 WPF 화면 안에서 Explorer Shell View가 정상적으로 표시된다.
- 주소 변경 또는 버튼 입력으로 폴더 이동이 가능하다.
- 파일/폴더 우클릭 시 Windows Shell 컨텍스트 메뉴가 표시된다.
- 기본적인 창 리사이즈와 포커스 동작에 치명적인 문제가 없다.

### 4.3 2단계: Multi Panel MVP

초기 MVP는 기능을 많이 넣기보다 “Explorer 같은 패널이 여러 개 있다”는 핵심 경험을 완성하는 데 집중한다.

포함 기능:

- 1패널, 2패널, 3패널, 4패널 레이아웃
- 사용자가 패널 수를 1~4개 범위에서 전환
- 각 패널에 Explorer Shell View 표시
- 각 패널별 주소 표시줄
- 뒤로, 앞으로, 상위 폴더 이동
- 기본 탭 기능
- 패널 간 파일 복사/이동
- Windows 우클릭 메뉴 유지
- 다크/라이트 기본 테마
- 마지막 상태 복원을 위한 패널/탭 상태 구조 준비

완료 기준:

- 사용자가 여러 폴더를 동시에 열고 비교할 수 있다.
- 각 패널은 독립적인 현재 경로와 탐색 히스토리를 가진다.
- 실행 중 1~4패널 전환 시 각 패널의 상태가 분리되어 유지된다.
- Windows Shell 메뉴와 기본 파일 작업 흐름이 유지된다.

#### 4.3.1 1~4패널 제어 정책

MVP에서는 자유로운 임의 분할 편집보다 안정적인 고정 프리셋을 우선한다. 패널 수는 1개부터 4개까지 제어 가능해야 하며, 각 패널은 독립적인 경로, 탭, 탐색 히스토리, 활성 상태를 가진다.

권장 레이아웃 프리셋:

| 패널 수 | 기본 레이아웃 | 목적 |
| --- | --- | --- |
| 1 | 단일 전체 패널 | 일반 탐색, 단일 작업 |
| 2 | 좌우 2분할 | 복사/이동, 비교 작업 |
| 3 | 좌측 1개 + 우측 상하 2개 | 기준 폴더 1개와 보조 폴더 2개 비교 |
| 4 | 2x2 그리드 | 여러 작업 위치 동시 관리 |

패널 수 전환 시 기존 패널 상태는 가능한 한 유지한다. 예를 들어 4패널에서 2패널로 줄이면 숨겨진 패널의 경로와 탭 상태를 메모리에 보관하고, 다시 4패널로 늘렸을 때 복원한다. 앱 종료 시에는 현재 보이는 패널뿐 아니라 숨겨진 패널 상태까지 저장할지 여부를 설정 정책으로 정한다.

초기 구현에서는 다음 범위를 MVP로 둔다.

- 패널 수 선택: 1, 2, 3, 4
- 활성 패널 표시
- 패널별 주소창과 탐색 버튼
- 패널별 탭 상태 유지
- 레이아웃 전환 시 Shell View 생성/해제 또는 숨김 처리 정책 검증

자유 드래그 리사이즈, 사용자 정의 분할 비율 저장, 복잡한 레이아웃 편집기는 MVP 이후로 미룬다.

### 4.4 3단계: IDE형 탭 드래그 도킹

Waffle Browse는 VS Code 같은 IDE 구조를 참고해 탭을 드래그하여 패널 구조를 쉽게 바꿀 수 있도록 한다. 목표는 복잡한 도킹 시스템 전체를 구현하는 것이 아니라, 파일 탐색 작업에 필요한 1x1, 1x2, 2x1, 2x2 구조를 빠르게 만드는 것이다.

지원 목표:

- 탭을 같은 패널 안에서 순서 변경
- 탭을 다른 패널로 이동
- 탭을 화면의 좌/우/상/하 도킹 영역으로 드롭해 새 분할 생성
- 1x1, 1x2, 2x1, 2x2 레이아웃 전환
- 빈 패널이 되면 자동 제거 또는 기본 경로 패널로 유지
- 현재 도킹 구조를 작업공간과 마지막 상태 복원에 포함

권장 구현 시점은 다음과 같다.

| 시점 | 할 일 | 이유 |
| --- | --- | --- |
| 프로젝트 시작 | 탭/패널/레이아웃 상태 모델을 도킹 가능한 구조로 설계 | 나중에 탭 드래그를 붙일 때 데이터 구조를 갈아엎지 않기 위해 |
| Shell Panel PoC 이후 | 실제 탭 드래그와 도킹 UI 구현 시작 | Shell View의 포커스, 리사이즈, HWND 이동 안정성을 먼저 확인해야 함 |
| 3단계 | 1x1, 1x2, 2x1, 2x2 고정 도킹 프리셋 완성 | 커스텀 기능보다 먼저 패널 조작 경험을 확정해야 하기 때문 |
| MVP 이후 | 자유 분할 비율, 복잡한 중첩 도킹, 플로팅 창 검토 | 구현 난도와 테스트 범위가 커지기 때문 |

따라서 이 기능은 처음 개발 시작부터 전체 구현하지 않는다. 대신 첫 프로젝트 구조를 만들 때 `TabState`, `PanelState`, `DockLayoutState` 같은 상태 모델을 먼저 분리해 둔다. 실제 드래그 앤 드롭 UI는 단일 Shell Panel과 2패널 PoC가 성공한 뒤 붙이는 것이 안전하다.

3단계에서는 다음 범위까지만 구현한다.

- 도킹 대상 표시: 좌, 우, 상, 하, 가운데
- 최대 패널 수: 4개
- 허용 레이아웃: 1x1, 1x2, 2x1, 2x2, 3패널 기본 프리셋
- 탭 이동 결과가 4패널을 초과하면 드롭 불가 표시
- 패널별 활성 탭과 현재 경로 유지

외부 도킹 라이브러리는 바로 채택하지 않고, 먼저 Waffle Browse에 필요한 제한된 도킹 모델을 직접 정의한다. `IExplorerBrowser`는 네이티브 HWND/COM 기반이므로 범용 도킹 라이브러리와 조합할 때 포커스, 클리핑, 리사이즈 문제가 생길 수 있다. 라이브러리 도입은 Shell Panel PoC와 2패널 검증 이후 별도 후보로 검토한다.

### 4.5 4단계: 도킹 레이아웃 상태 안정화

IDE형 탭 드래그 도킹이 완료된 뒤에는 커스텀 기능을 바로 붙이기 전에 레이아웃 상태 저장과 복원을 안정화한다. 이 단계는 사용자 기능을 늘리는 단계라기보다, 이후 작업공간과 검색 기능이 안정적으로 올라갈 수 있게 하는 기반 단계다.

포함 기능:

- 현재 도킹 구조 저장
- 패널별 탭 목록 저장
- 패널별 활성 탭과 현재 경로 저장
- 앱 재시작 시 마지막 도킹 구조 복원
- 숨겨진 패널 또는 닫힌 패널의 상태 보존 정책 확정
- 잘못된 경로, 삭제된 폴더, 접근 권한 없는 폴더 복원 실패 처리

완료 기준:

- 1x1, 1x2, 2x1, 2x2 상태가 앱 재시작 후 복원된다.
- 각 패널의 탭과 현재 경로가 분리되어 저장된다.
- 복원 실패 경로가 있어도 앱 전체 실행이 실패하지 않는다.
- 이 단계까지 완료한 뒤 Waffle Browse 고유 기능을 추가한다.

### 4.6 5단계: Waffle Browse 고유 기능

이 단계부터는 단순한 멀티 패널 탐색기를 넘어 Waffle Browse만의 사용성을 만든다. IDE형 탭 드래그 도킹과 레이아웃 상태 복원이 완료된 뒤에 커스텀 기능을 붙이는 순서가 맞다.

포함 기능:

- 빠른 파일/폴더 검색
- 현재 패널 안 필터 검색
- 전체 열린 패널 대상 검색
- 검색 결과 패널
- 검색 결과에서 파일 위치 열기
- 검색 결과에서 파일 작업 실행
- 즐겨찾기 폴더
- 작업공간 저장/불러오기
- 레이아웃 프리셋
- 자주 쓰는 폴더 묶음 열기
- 패널별 이름 지정

작업공간 예시:

- 개발
- 다운로드 정리
- 사진 백업
- 업무 문서

작업공간은 레이아웃, 패널 구성, 탭, 경로, 패널 이름을 저장하고 한 번에 복원하는 기능으로 정의한다.

검색은 Waffle Browse의 이름과 직접 연결되는 핵심 기능이다. 다만 Shell Panel 안정화, 1~4패널 제어, IDE형 탭 도킹, 레이아웃 복원이 완료되기 전에는 검색을 과하게 확장하지 않는다.

초기 검색은 열린 패널과 현재 폴더 중심으로 제한하고, 인덱싱 기반 전체 디스크 검색은 후순위로 둔다.

### 4.7 6단계: Polish 및 배포

MVP와 핵심 기능이 안정화된 뒤 사용자 경험과 배포 방식을 다듬는다.

포함 기능:

- 테마 고도화

- 설정 화면
- 설치형 배포

## 5. MVP 이후 확장 기능

다음 기능은 초기 버전에 포함하지 않고, Shell Panel과 작업공간 기능이 안정화된 뒤 검토한다.

- 미리보기 패널
- 폴더 크기 계산
- 파일 목록 CSV/TXT 내보내기
- 색상 필터
- 태그/메모
- portable 모드
- 설정 백업/복원
- 단축키 커스터마이징

특히 태그/메모와 색상 필터는 자체 메타데이터 저장소가 필요할 수 있으므로, 단순 UI 기능보다 데이터 설계의 영향이 크다. MVP 범위에서는 제외하는 것이 적절하다.

## 6. 우선순위 정리

| 단계 | 이름 | 핵심 목표 | 성공 기준 |
| --- | --- | --- | --- |
| 0 | 프로젝트 생성 | WPF 기반 앱 골격과 도킹 가능한 상태 모델 준비 | 실행 가능한 기본 앱과 탭/패널/레이아웃 모델 |
| 1 | Shell Panel PoC | Explorer View 호스팅 검증 | 경로 이동과 우클릭 메뉴 동작 |

| 3 | IDE형 탭 드래그 도킹 | 탭 드래그로 패널 구조 변경 | 1x1, 1x2, 2x1, 2x2 도킹 |
| 4 | 도킹 레이아웃 상태 안정화 | 도킹 구조와 탭/경로 복원 기반 완성 | 앱 재시작 후 도킹 구조 복원 |
| 5 | Waffle Browse 고유 기능 | 커스텀 작업 흐름 추가 | 작업공간, 즐겨찾기, 검색 |
| 6 | Polish | 제품화 | 테마, 단축키, 설정, 배포 |

## 7. 의사결정 기준

프로젝트의 핵심 결정은 다음과 같이 정리한다.

> Waffle Browse의 파일 표시 영역은 Windows Explorer Shell을 사용하고, Waffle Browse는 멀티 패널, 탭, 검색, 작업공간을 제공하는 래퍼 앱으로 만든다.

따라서 초기 개발에서는 새 파일 목록 UI를 직접 만드는 방향을 피하고, Windows Shell View를 앱 내부에 안정적으로 호스팅하는 데 집중해야 한다. 이 검증이 끝난 뒤에야 멀티 패널, 탭, 작업공간, 검색 기능을 쌓는 것이 안전하다.

## 8. 다음 작업 제안

바로 이어서 진행할 작업은 다음 순서가 적절하다.

1. WPF 프로젝트 생성
2. 탭/패널/도킹 레이아웃 상태 모델 설계
3. `IExplorerBrowser` interop PoC 구현
4. 단일 Shell Panel에서 경로 이동과 우클릭 메뉴 검증
5. 2패널 Shell View 동시 호스팅 검증
6. 3패널, 4패널 레이아웃 확장 및 스트레스 테스트
7. 검증 결과를 기준으로 Multi Panel MVP 설계 확정
8. 탭 드래그 기반 1x1, 1x2, 2x1, 2x2 도킹 구현
9. 도킹 레이아웃과 마지막 상태 복원 안정화
10. Waffle Browse 고유 기능 설계 확정

이 중 가장 중요한 작업은 3번이다. `IExplorerBrowser`가 앱 내부 패널에서 안정적으로 동작한다는 사실이 확인되어야 이후 개발 범위를 현실적으로 확정할 수 있다. 다만 2번의 상태 모델 설계는 처음부터 해둬야 한다. 탭과 패널 상태가 단순 UI 컨트롤에 묶이면, 나중에 IDE형 도킹을 붙일 때 구조 변경 비용이 커진다.

## 9. 최종 개발 순서 검토

현재 계획은 다음 순서로 진행하는 것이 타당하다.

1. WPF 프로젝트 생성
2. 탭/패널/도킹 레이아웃 상태 모델 설계
3. 단일 `IExplorerBrowser` 패널 PoC
4. 2패널 동시 호스팅 PoC
5. 3패널/4패널 레이아웃 확장
6. 패널별 주소창, 탐색 버튼, 기본 탭 상태 구현
7. 탭 드래그 기반 1x1, 1x2, 2x1, 2x2 도킹 구현
8. 도킹 레이아웃과 마지막 경로 자동 복원
9. 작업공간, 즐겨찾기, 패널 이름, 레이아웃 프리셋
10. 빠른 검색과 검색 결과 패널
11. 테마, 단축키, 설정, 설치/portable 배포

이 순서는 그대로 개발을 진행해도 된다. 단, 3번부터 5번까지는 기능 개발이 아니라 기술 검증으로 취급해야 한다. 특히 4패널 동시 실행에서 포커스, 리사이즈, 메모리 사용량, Shell COM 리소스 해제가 안정적인지 확인한 뒤에 탭 드래그 도킹을 붙이는 것이 안전하다. 이후 7번과 8번까지 완료한 뒤 Waffle Browse 고유 기능인 작업공간, 즐겨찾기, 검색을 추가한다.

IDE형 탭 도킹은 처음부터 완성 기능으로 만들면 안 된다. 하지만 처음부터 고려하지 않으면 나중에 구조를 크게 바꿔야 하므로, 개발 시작 시점에는 상태 모델과 저장 포맷만 도킹 가능한 형태로 잡고, 실제 드래그 UI는 Shell Panel 검증 이후에 구현한다.

개발 시작 전 최종 확인 기준은 다음과 같다.

- 단일 패널에서 `IExplorerBrowser` 렌더링, 경로 이동, 우클릭 메뉴가 동작한다.
- 2패널 이상에서 각 패널의 포커스와 경로 상태가 분리된다.
- 1~4패널 전환 시 레이아웃이 깨지지 않는다.
- 탭 드래그로 1x1, 1x2, 2x1, 2x2 구조를 만들 수 있다.
- 패널 제거/숨김/복원 시 Shell View 리소스가 누수되지 않는다.
- Shell View 내부 기능을 과도하게 커스터마이징하지 않고, Waffle Browse의 기능은 앱 레벨에서 붙인다.

위 기준을 만족하면 이 계획은 현실적으로 진행 가능하다. 실패 지점이 나온다면 Explorer Shell View를 유지할지, 자체 파일 목록 UI를 일부 도입할지 그 시점에 다시 판단한다.
