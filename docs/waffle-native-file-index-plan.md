# Waffle Native File Index Architecture

## 목적과 현재 기준선

Waffle Browse 검색은 외부 검색 엔진 없이 Waffle이 소유한 로컬 인덱스로 파일 이름과 경로를 검색한다.

조사에서 확인한 기존 동작은 다음과 같다.

- 기존 외부 SDK/PInvoke 검색 경로는 제거되었다.
- 검색 입력은 200ms 디바운스되고 검색 결과 화면이 활성 상태일 때 1초마다 재조회한다.
- 전체 인덱스/현재 폴더 범위, 최대 1,000개, 폴더 우선 정렬과 이름·경로·수정일·크기 정렬이 있다.
- `LatestSearchRequestCoordinator`가 이전 요청을 취소하고 늦게 도착한 응답을 폐기한다.
- `waffle-search:` 탭 상태와 기존 `search-ms:` 상태 마이그레이션이 있다.
- 일반 폴더는 기존 `IExplorerBrowser` 호스트로 계속 렌더링한다.

자체 엔진은 Waffle이 소유하는 인덱스 구축, 영속화, 변경 추적 수명주기를 가진다.

## 단계 1: 완성된 수직 단계

이 저장소에는 다음 경계가 구현되어 있다.

```text
SearchResultsView
  -> LatestSearchRequestCoordinator
  -> WaffleFileSearchProvider
      -> FileSearchIndex
      -> IFileIndexSource
      -> IFileIndexStore
```

Waffle 자체 공급자가 유일한 검색 경로다. 첫 스냅샷이 준비되기 전에는 구축 상태를 UI에 표시하고 부분 또는 외부 결과로 폴백하지 않는다.

현재 Waffle 공급자는 독립적으로 동작하는 첫 구현으로 다음을 제공한다.

- 고정 NTFS 드라이브를 UI 스레드 밖에서 재귀 열거한다.
- 파일명, 전체 경로, 부모 경로, 파일/폴더 종류, 크기, 수정 시각을 인덱싱한다.
- 이름 또는 경로의 대소문자 무시 부분 일치와 `*`/`?` 단순 와일드카드를 지원한다.
- 전체/현재 폴더 범위, 최대 1,000개, 폴더 우선 및 기존 정렬 옵션을 지원한다.
- `ReaderWriterLockSlim`으로 동시 검색과 갱신에 일관된 경계를 둔다.
- `FileSystemWatcher`로 생성·변경·삭제·이름 변경을 반영한다. 폴더 이름 변경은 모든 하위 인덱스 경로를 함께 이동한다.
- 초기 재구축 중 이벤트를 버퍼링한 뒤 새 스냅샷에 적용한다.
- 변경 감시 버퍼 오버플로는 `NeedsRebuild`로 전환하고 전체 재구축을 예약한다.
- 버전이 있는 JSON 스냅샷을 임시 파일에 쓴 뒤 원자적으로 교체한다. 손상/형식 불일치는 안전하게 재구축한다.
- 상태 모델에 `Empty`, `Loading`, `Ready`, `Rebuilding`, `NeedsRebuild`, `Failed`와 세대, 항목 수, 완료 시각, 볼륨 체크포인트를 둔다.

이 단계는 기능 경계와 복구 의미를 검증하기 위한 수직 단계다. 전체 디스크 재귀 열거 및 JSON은 수백만 항목의 최종 성능 설계가 아니다. 다음 단계에서 `IFileIndexSource`와 `IFileIndexStore` 구현을 교체하되 검색/UI 계약은 유지한다.

## 단계 2: NTFS MFT 초기 스냅샷

`NtfsMftIndexSource : IFileIndexSource`를 별도 Windows 구현으로 추가한다.

1. `\\.\C:` 형태의 볼륨 핸들을 열고 볼륨 ID, 파일시스템, 직렬 번호를 기록한다.
2. `FSCTL_ENUM_USN_DATA`/`MFT_ENUM_DATA_V0`로 MFT 레코드를 열거한다.
3. `USN_RECORD_V2/V3`의 파일 참조 번호(FRN), 부모 FRN, Unicode 이름, 속성을 파싱한다.
4. FRN 그래프를 루트부터 해석해 전체 경로를 만든다. 고아·순환·부모 누락 레코드는 격리하고 경고 수에 포함한다.
5. 파일 ID 기반 조회로 크기와 수정 시각을 보강한다. 조회 실패 항목은 이름/경로 검색에는 남기되 메타데이터를 `null`로 둔다.
6. 한 볼륨의 완성된 세대를 만든 뒤에만 활성 스냅샷과 교체한다. 취소 또는 볼륨 분리 시 이전 세대를 유지한다.

네이티브 구조체는 명시적 레이아웃과 고정 폭 정수를 사용하고, 레코드 길이·이름 오프셋·버퍼 경계를 매 레코드마다 검증한다. 앱은 x64이지만 구조 파서는 x86/x64 포인터 크기에 의존하지 않게 작성한다. 모든 문자열은 Unicode다.

## 단계 3: USN Change Journal 증분 처리

초기 스냅샷이 완료되면 `FSCTL_QUERY_USN_JOURNAL` 결과의 Journal ID와 Next USN을 `FileIndexCheckpoint`에 저장한다. 재시작 시 다음 순서를 따른다.

1. 저장된 볼륨 ID가 현재 볼륨과 같은지 확인한다.
2. Journal ID가 같고 저장된 Next USN이 현재 유효 범위 안이면 `FSCTL_READ_USN_JOURNAL`로 이어 읽는다.
3. create/delete/data/basic-info 변경은 FRN 단위 upsert/delete로 합친다.
4. old-name/new-name rename 이유를 FRN과 USN 순서로 짝지어 한 트랜잭션으로 적용한다. 짝이 없으면 해당 FRN을 재조회한다.
5. 배치를 새 불변 세대로 게시한 뒤에만 체크포인트를 원자적으로 저장한다.

다음 조건은 전체 볼륨 재구축으로 전환한다.

- 저널 미존재 또는 비활성화
- Journal ID 변경
- 저장 USN이 LowestValidUsn보다 오래됨(랩/삭제)
- 네이티브 레코드 손상 또는 순서 불일치
- 볼륨 ID 변경, 분리, 재포맷
- 영속 스냅샷과 체크포인트 세대 불일치

검색은 재구축 중에도 마지막 정상 세대를 읽는다. 새 세대가 완성되기 전에는 부분 결과를 게시하지 않는다.

## 단계 4: 권한과 프로세스 경계

전체 볼륨 MFT/USN 접근이 일반 사용자 토큰에서 거부되면 앱 전체를 관리자 권한으로 실행하지 않는다. `Waffle.Browse.Indexer` Windows 서비스 또는 최소 권한 헬퍼를 별도 배포 단위로 둔다.

- 설치/업데이트/제거 경계를 앱과 분리한다.
- 로컬 named pipe를 사용하고 현재 사용자 SID 또는 허용된 로컬 사용자 ACL을 검증한다.
- 요청/응답 프레임에 버전과 최대 크기(초기 64 KiB)를 둔다.
- 검색어를 명령줄, HTTP URL, 이벤트 로그에 전달하지 않는다.
- 서비스는 원시 사용자 쿼리를 실행하지 않고 정규화된 검색 계약만 받는다.
- 응답 결과 수는 1,000개, 경로 길이와 쿼리 길이는 명시적으로 제한한다.
- 서비스가 없거나 권한이 거부되면 마지막 정상 Waffle 스냅샷을 유지하거나 사용자 모드 재귀 인덱싱으로 안전하게 폴백한다.

## 단계 5: 비 NTFS와 네트워크

NTFS가 아닌 로컬 볼륨은 현재 `RecursiveFileIndexSource`를 유지하고 `FileSystemWatcher`/`ReadDirectoryChangesW`로 추적한다. 네트워크 공유는 사용자가 명시적으로 추가한 루트만 인덱싱하고, 연결 끊김·이벤트 신뢰도·비용 때문에 주기적 재검증을 병행한다. 볼륨별 공급원과 체크포인트가 독립적이므로 한 볼륨 실패가 다른 볼륨 세대를 폐기하지 않게 한다.

## 영속 저장소 후속 결정

JSON 스냅샷은 포맷/복구 테스트에는 충분하지만 대형 인덱스의 시작 시간, 파일 크기, 쓰기 증폭이 크다. 외부 패키지를 추가하기 전에 다음을 벤치마크한다.

- Waffle 소유의 버전 있는 이진 스냅샷 + 메모리 매핑
- Windows inbox API를 이용한 ESE
- SQLite를 사용할 경우 네이티브 배포 크기, 라이선스 고지, 업데이트 전략

선택 기준은 1천만 항목 초기 로드 시간, 이름 토큰 조회 지연, 배치 갱신 비용, 충돌 후 복구, 배포 크기다.

## 검증 기준

- Core 단위 테스트: 범위, 와일드카드, 1,000개 제한, 폴더 우선/정렬, 생성·삭제·폴더 이름 변경, 인덱스 상태.
- 통합 테스트: 임시 로컬 폴더의 초기 구축, JSON 저장, 생성·이름 변경·삭제 이벤트 반영, 손상 JSON 자동 재구축.
- App 테스트: 이전 요청 취소와 늦은 응답 폐기, 검색 범위 설정 영속화.
- 회귀: 일반 폴더 `IExplorerBrowser`, 도킹, 탭, 테마 테스트 전체 통과.

NTFS 네이티브 단계에서는 별도 테스트 볼륨/임시 VHDX로 저널 생성, 랩, 삭제, 볼륨 분리, 권한 거부, 레코드 손상을 재현하고 관리자 서비스 설치 여부와 무관한 사용자 모드 테스트를 유지한다.
